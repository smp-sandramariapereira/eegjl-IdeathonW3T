# Classroom Integration

## 1. Discovery

- Activity: identify the academic evidence problem.
- Decision: focus on verifiable proof, not on storing student records.
- Evidence produced: problem statement and scope note.
- Teacher role: frame the challenge and enforce privacy rules.
- Student role: describe the artifact and its verification need.
- Ethereum use: not yet required.

## 2. Ideation

- Activity: compare solution options.
- Decision: choose a minimal proof-based approach.
- Evidence produced: shortlist of alternatives and rationale.
- Teacher role: challenge assumptions and keep the scope educational.
- Student role: propose use cases and identify non-personal identifiers.
- Ethereum use: only if it answers a real verification need.

## 3. Solution

- Activity: define the user journey for issuing and verifying evidence.
- Decision: separate content from proof.
- Evidence produced: flow description and draft record format.
- Teacher role: validate pedagogical clarity.
- Student role: design the sequence of actions.
- Ethereum use: planned as the verification layer, not as a content store.

## 4. Technology

- Activity: decide the technical stack.
- Decision: use Sepolia, Solidity, Hardhat, TypeScript, and ethers for the planned pilot.
- Evidence produced: stack note and technical justification.
- Teacher role: review whether blockchain is justified.
- Student role: explain why a conventional database is or is not enough.
- Ethereum use: planned, under design, and testnet-only.

## 5. MVP

- Activity: define the smallest viable proof flow.
- Decision: limit the MVP to hashes, identifiers, issuer, and timestamp.
- Evidence produced: MVP scope document.
- Teacher role: prevent scope creep.
- Student role: prioritize the minimum useful flow.
- Ethereum use: limited to the planned proof record.

## 6. Architecture

- Activity: map components and data boundaries.
- Decision: keep all personal data offchain.
- Evidence produced: architecture diagram and privacy model.
- Teacher role: confirm separation of onchain and offchain data.
- Student role: document each component clearly.
- Ethereum use: used only for the proof layer.

## 7. Development

- Activity: build the local prototype and supporting scripts.
- Decision: start with the smallest verifiable contract behavior.
- Evidence produced: code, tests, and execution notes.
- Teacher role: inspect evidence and test results.
- Student role: implement and verify the planned behavior.
- Ethereum use: local and testnet validation only.

## 8. Pitch

- Activity: present the pilot and its limitations.
- Decision: communicate honestly what is planned versus completed.
- Evidence produced: presentation, repository links, and demo notes.
- Teacher role: evaluate clarity and honesty.
- Student role: explain the privacy model and technical architecture.
- Ethereum use: shown as a planned pilot or testnet validation, never as a production claim.
